Introduction and Advanced Concepts of File Attributes

14 June, 2026 Linux Administration Jobaer Hossain

লিনাক্স ফাইল সিস্টেমের নিরাপত্তা এবং ফাইল ম্যানেজমেন্ট নিয়ে আলোচনা করার সময় আমরা সাধারণত Read, Write এবং Execute পারমিশন (যা chmod কমান্ড দিয়ে নিয়ন্ত্রণ করা হয়) নিয়ে বিস্তারিত পড়াশোনা করি। কিন্তু অনেক সময় এন্টারপ্রাইজ প্রোডাকশন এনভায়রনমেন্টে এমন কিছু পরিস্থিতি তৈরি হয় যেখানে প্রচলিত ফাইল পারমিশন মেকানিজম একা সিস্টেমকে শতভাগ সুরক্ষা দিতে ব্যর্থ হয়। উদাহরণস্বরূপ, লিনাক্সের সর্বোচ্চ ক্ষমতাশালী Root User চাইলে যেকোনো Read-only ফাইলকে এক সেকেন্ডে ডিলিট বা মডিফাই করে দিতে পারেন।

তাহলে উপায় কী? কার্নেল লেভেলে ফাইলকে আরও গভীরভাবে লক করার জন্য লিনাক্স ফাইল সিস্টেম (যেমন ext4, XFS) আরেকটি লেয়ার ব্যবহার করে, যাকে বলা হয় File Attributes। এই বিশেষ এট্রিবিউটের মাধ্যমে একটি ফাইলকে এমনভাবে প্রটেক্ট করা সম্ভব যে স্বয়ং রুট ইউজারও ফাইলটি ডিলিট, রিনেম বা এডিট করতে পারবেন না যতক্ষণ না সেই এট্রিবিউট লক সরানো হচ্ছে। আজকের এই ডিপ-ডাইভ টিউটোরিয়ালে আমরা লিনাক্স ফাইল এট্রিবিউটের ইন্টারনাল মেকানিজম, এর বিভিন্ন ফ্ল্যাগ, chattr ও lsattr কমান্ডের প্র্যাকটিক্যাল কনফিগারেশন এবং ট্রাবলশুট নিয়ে বিস্তারিত আলোচনা করবো।

১. File Permissions বনাম File Attributes

কাজ শুরু করার আগে এই দুটি কনসেপ্টের মধ্যকার মূল টেকনিক্যাল ও লজিক্যাল পার্থক্যটি বোঝা অত্যন্ত জরুরী:

  • File Permissions (chmod): এটি ওএস লেভেলে ডিফাইন করা একটি সিকিউরিটি ফিল্টার, যা নির্দিষ্ট ইউজার বা গ্রুপ অ্যাকাউন্টের অ্যাক্সেস রেস্ট্রিকশন (rwx) নিয়ন্ত্রণ করে। তবে এই পারমিশনের ফাইল বাউন্ডারি root (UID=0) ইউজারের কাছে প্রযোজ্য নয়। রুট ইউজার রাইট পারমিশন ছাড়াও যেকোনো ফাইল ওভাররাইট করতে পারেন।
  • File Attributes (chattr): এটি সরাসরি ফাইল সিস্টেমের লো-লেভেল মেটাডাটা বা Inode Block-এর ভেতরে একটি ফ্ল্যাগ বা প্রপার্টি হিসেবে লক হয়ে থাকে। এটি ইউজারের ওপরে কাজ করে না, এটি সরাসরি ফাইলের আচরণের (Behavior) ওপর রেস্ট্রিকশন আরোপ করে। ফলে রুট ইউজারও এই লকিং মেকানিজমের বাউন্ডারি ব্রেক করতে পারেন না।

রিয়েল-লাইফ সিনারিও (The Bank Safe Box Analogy)

পারমিশন এবং এট্রিবিউটের প্র্যাকটিক্যাল পার্থক্য বুঝতে একটি ব্যাংকের লকার বা সেফ বক্সের কথা চিন্তা করুন:

  • File Permissions (লকার রুমের সিকিউরিটি গার্ড): এটি হলো ব্যাংকের গেটে থাকা সিকিউরিটি গার্ড। সে সাধারণ মানুষকে ভেতরে ঢুকতে বাধা দেয়, কিন্তু ব্যাংকের মেইন ডিরেক্টর বা ম্যানেজার (Root User) আসলে তাকে সসম্মানে ভল্ট রুমে ঢুকতে পারমিশন দিয়ে দেয়।
  • File Attributes (ডিজিটাল টাইম-লক মেকানিজম): এটি হলো ভল্টের দরজার ভেতরে বিল্ট-ইন থাকা একটি মেকানিক্যাল বা ডিজিটাল লক, যা প্রোগ্রাম করা আছে যে আগামী ২৪ ঘণ্টা এই দরজা খোলা যাবে না। এখন ব্যাংকের ম্যানেজার নিজে এসে চাবি ঘোরালেও দরজা খুলবে না, কারণ লকটি সরাসরি দরজার ফিজিক্যাল মেকানিজমের অংশ। এটিই হলো ফিজিক্যাল File Attributes।

২. গুরুত্বপূর্ণ কোর File Attributes এবং এদের ফ্ল্যাগ সমূহ

লিনাক্স কার্নেল বেশ কিছু সুনির্দিষ্ট ফাইল এট্রিবিউট সাপোর্ট করে। কমান্ড লাইনে এদেরকে ছোট হাতের বা বড় হাতের একক লেটার বা ফ্ল্যাগ (Flags) দিয়ে প্রকাশ করা হয়। সবচেয়ে গুরুত্বপূর্ণ ফ্ল্যাগগুলো নিচে দেওয়া হলো:

ক. Immutable Attribute (`i` ফ্ল্যাগ)

এটি লিনাক্সের সবচেয়ে শক্তিশালী এবং বহুল ব্যবহৃত সিকিউরিটি এট্রিবিউট। কোনো ফাইলে i ফ্ল্যাগ সেট করলে সেই ফাইলটি সম্পূর্ণ অপরিবর্তনশীল বা Immutable হয়ে যায়।

  • এই ফাইলটি কোনো ইউজার (এমনকি রুট) ডিলিট বা রিনেম করতে পারবেন না।
  • ফাইলের ভেতরে নতুন কোনো ডাটা রাইট করা যাবে না এবং নতুন কোনো হার্ড লিঙ্ক বা সফট লিঙ্ক তৈরি করা যাবে না।
  • Use Case: সিস্টেমের অতি সংবেদনশীল ফাইল যেমন— /etc/passwd, /etc/shadow বা নেটওয়ার্ক কনফিগারেশন ফাইল /etc/resolv.conf হ্যাকারদের হাত থেকে বা এক্সিডেন্টাল ডিলিশন থেকে বাঁচাতে এটি ব্যবহার করা হয়।

খ. Append-Only Attribute (`a` ফ্ল্যাগ)

এই এট্রিবিউটটি সেট করলে ফাইলটি ডিলিট বা ওভাররাইট করা সম্পূর্ণ ব্লক হয়ে যায়, তবে ফাইলের একদম শেষ লাইনের পর নতুন ডাটা যোগ (Append) করার অনুমতি থাকে।

  • ফাইলের ভেতরের পুরনো ডাটা কেউ মুছতে বা এডিট করতে পারে না।
  • শুধুমাত্র নতুন ডাটা লাইনের শেষে ইনজেক্ট করা যায়।
  • Use Case: এন্টারপ্রাইজ সিস্টেমের Log Files (যেমন— /var/log/secure বা /var/log/messages)। হ্যাকাররা সার্ভারে ঢুকে যেন তাদের ট্রেইল বা লগ মুছে প্রমাণ গায়েব করতে না পারে, তার জন্য লগ ফাইলে a ফ্ল্যাগ দিয়ে লক করা হয়।

গ. অন্যান্য স্পেশাল এট্রিবিউটস (`s`, `A`, `d`)

  • `s` (Secure Deletion): এই ফ্ল্যাগটি কোনো ফাইলে অন থাকলে, ফাইলটি ডিলিট করার সাথে সাথে হার্ডডিস্কের ওই নির্দিষ্ট ব্লক বা সেক্টরে জিরো (0) রাইট হয়ে যায়। ফলে ডাটা রিকভারি সফটওয়্যার দিয়েও সেই ফাইল আর কখনো ফিরিয়ে আনা সম্ভব হয় না।
  • `A` (No Atime Updates): লিনাক্সে কোনো ফাইল রিড বা ওপেন করলেই তার মেটাডাটায় Access Time (atime) আপডেট হয়, যা ডিস্কে অতিরিক্ত রাইট অপারেশন তৈরি করে। এই ফ্ল্যাগটি অন রাখলে ওএস atime রাইট করা বন্ধ রাখে, যা ল্যাপটপ বা সার্ভারের SSD-র পারফরম্যান্স ও আয়ু বহুগুণ বাড়িয়ে দেয়।
  • `d` (No Dump): লিনাক্সের ট্র্যাডিশনাল dump ব্যাকআপ টুল রান করার সময় যে ফাইলগুলোতে এই ফ্ল্যাগ সেট করা থাকে, সেগুলোকে ব্যাকআপ ইমেজ থেকে স্কিপ করা হয়।

৩. এক নজরে ফাইল এট্রিবিউট রেফারেন্স টেবিল

Attribute Flag Technical Name Root User Overwrite Capability Permitted Operations Scope Primary Tactical Security Use Case
i Immutable Strictly Blocked Read-only (No write, delete, rename, link creation). Protecting core OS configs from Ransomware and rootkits.
a Append-Only Strictly Blocked Only text/data insertion at the absolute end of the file. Securing auditing log files from log tampering attacks.
A No atime update Allowed All normal ops, but suppresses access time metadata disk writes. Optimizing high-frequency I/O database storage and SSD lifespan.
s Secure Deletion Allowed Wipes the secondary storage blocks completely with zeros upon deletion. Handling top-secret cryptographic keys or intellectual property.

৪. প্র্যাকটিক্যাল CLI অপারেশন: chattr ও lsattr কমান্ডের ব্যবহার

ফাইল এট্রিবিউট চেক এবং কনফিগার করার জন্য লিনাক্সে দুটি ডেডিকেটেড কমান্ড রয়েছে: lsattr (List Attributes) এবং chattr (Change Attributes)। এই অপারেশনগুলো রান করতে বাধ্যতামূলকভাবে root বা sudo প্রিভিলেজ প্রয়োজন হয়।

ধাপ ১: বর্তমান ফাইল এট্রিবিউট চেক করা

একটি নির্দিষ্ট ফাইলের লাইভ এট্রিবিউট স্ট্যাটাস দেখার কমান্ড:

$ lsattr important_config.conf

# আউটপুট দেখতে নিচের মতো ফ্ল্যাট ড্যাশ স্ট্রিং হবে:
# --------------e------- important_config.conf

ব্যাখ্যা: এখানে e মানে ফাইলটি Extents ব্যবহার করছে (আধুনিক ফাইল সিস্টেমের ডিফল্ট আচরণ)। বর্তমানে কোনো সিকিউরিটি লক অন নেই।

ধাপ ২: ফাইলকে Immutable বা আন-ডিলিটেবল করা (+i)

ফাইল সিস্টেমে কাস্টম এট্রিবিউট যোগ করার জন্য প্লাস (+) সাইন ব্যবহার করা হয়:

$ sudo chattr +i important_config.conf

এবার পুনরায় lsattr দিয়ে ভেরিফাই করুন:

$ lsattr important_config.conf
# আউটপুট: ----i---------e------- important_config.conf

ধাপ ৩: টেস্ট ডিলিশন (Testing the Lock)

আসুন আমরা প্রমাণ করে দেখি যে লকটি আসলেই কাজ করছে কি না। রুট প্রিভিলেজ নিয়ে ফাইলটি ফোর্সফুলি ডিলিট করার চেষ্টা করুন:

$ sudo rm -rf important_config.conf

# কার্নেল সরাসরি কমান্ডটি রিজেক্ট করে এরর মেসেজ দেবে:
# rm: cannot remove 'important_config.conf': Operation not permitted

ভেরিফিকেশন চেক: ওএস পারমিশন ডেনাইড বলেনি, বলেছে Operation not permitted। এর মানে ফাইল সিস্টেমের লো-লেভেল ড্রাইভার নিজেই এই প্রসেস ব্লক করে দিয়েছে।

ধাপ ৪: লক রিমুভ বা আনলক করা (-i)

ফাইল থেকে কোনো এট্রিবিউট মুছে ফেলার জন্য মাইনাস (-) সাইন ব্যবহার করা হয়:

# ফাইলটি পুনরায় এডিট বা ডিলিট করার উপযোগী করতে
$ sudo chattr -i important_config.conf

৫. Troubleshooting এবং প্রফেশনাল সিকিউরিটি টিপস

  • The Ransomware or Malware Rootkit Trap: সাইবার সিকিউরিটি অডিটের সময় অনেক সময় দেখা যায় হ্যাকাররা সার্ভারে ঢুকে কোনো ব্যাকডোর স্ক্রিপ্ট (Backdoor) বা ম্যালওয়্যার ফাইল তৈরি করে সেটিতে +i বা Immutable ফ্ল্যাগ সেট করে রেখে দেয়। এর ফলে সিস্টেম এডমিন সার্ভার ক্লিন করার সময় rm -rf দিয়েও ওই ক্ষতিকারক ফাইল ডিলিট করতে পারেন না এবং বিভ্রান্তিতে পড়েন।
    সমাধান মেথড: যদি রুট ইউজার হয়েও কোনো ফাইল ডিলিট করতে না পারেন, তবে সিএসএস বা ফাইল পারমিশন চেকিংয়ে সময় নষ্ট না করে সরাসরি lsattr রান করুন। যদি i বা a ফ্ল্যাগ দেখতে পান, তবে প্রথমে chattr -i filename দিয়ে লক রিলিজ করুন এবং তারপর ফাইলটি ডিলিট করুন।
  • Log Rotation Breaking due to `+a` flag: আপনি আপনার সিস্টেমের সিকিউরিটি বাড়াতে /var/log/nginx/access.log ফাইলে chattr +a সেট করে দিলেন যাতে কেউ লগ মুছে না ফেলে। কিন্তু কিছুদিন পর দেখলেন আপনার Logrotate Service ক্র্যাশ করেছে এবং নতুন লগ তৈরি হচ্ছে না।
    কারণ: লিনাক্সের লগ-রোটেশন সার্ভিসটি পুরনো লগকে কমপ্রেস করে রিনেম (Rename) করতে চায় এবং একটি নতুন ব্ল্যাঙ্ক লগ ফাইল তৈরি করে। যেহেতু +a ফ্ল্যাগ রিনেম বা ফাইল রিপ্লেসমেন্ট ব্লক করে, তাই সার্ভিসটি ফেইল করেছে।
    সমাধান: লোটেশন স্ক্রিপ্টের কনফিগারেশনে (/etc/logrotate.d/nginx) prerotate এবং postrotate ডিরেক্টিভ ব্যবহার করে স্ক্রিপ্ট এক্সিকিউশনের ঠিক আগে ক্ষণস্থায়ীভাবে chattr -a এবং কাজ শেষে পুনরায় chattr +a অটোমেট করে দিন।
  • Directory Level Inheritance: যদি আপনি কোনো নির্দিষ্ট ডিরেক্টরি এবং তার ভেতরের সমস্ত সাব-ফাইল ও ফোল্ডারকে একসাথে সাবনেটেড বা ইমিউটেবল করতে চান, তবে রিকার্সিভ (Recursive) ফ্ল্যাগ -R ব্যবহার করুন: 
    $ sudo chattr -R +i /home/jobaer/protected_directory/

উপসংহার (Conclusion)

Linux File Attributes এবং chattr/lsattr মেকানিজম অপারেটিং সিস্টেমের ফিজিক্যাল ও লজিক্যাল লেয়ারের এক অত্যন্ত শক্তিশালী ডিফেন্স ইন ডেপথ (Defense-in-depth) সিকিউরিটি ফ্রেমওয়ার্ক। এটি প্রথাগত chmod পারমিশন মডেলের সীমাবদ্ধতা ভেঙে রুট ইউজারের ভুল অপারেশন বা ম্যালিসিয়াস প্রসেস থেকে কোর ওএস ফাইল এবং অডিট লগকে শতভাগ অক্ষুণ্ণ রাখতে সহায়তা করে। একজন দক্ষ এবং প্রো-লেভেল লিনাক্স সিস্টেম অ্যাডমিনিস্ট্রেটর হিসেবে এই ইনোড-লেভেল লকিং প্রপার্টিগুলো বুঝতে পারা আপনার ট্রাবলশুটিং ও ইনফ্রাস্ট্রাকচার হার্ডেনিং (System Hardening) ক্ষমতা বহুগুণ বাড়িয়ে দেবে, যা আপনাকে রিয়েল-লাইফ ডাটা সেন্টারকে যেকোনো ক্রিটিক্যাল র‍্যানসমওয়্যার বা ডেস্ট্রাক্টিভ অ্যাটাক থেকে সম্পূর্ণ নিরাপদ রাখতে সাহায্য করবে।
