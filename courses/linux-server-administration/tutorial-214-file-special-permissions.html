Introduction, Concepts and Deep Dive into File Special Permissions

14 June, 2026 Linux Administration Jobaer Hossain

লিনাক্স ফাইল সিস্টেম সিকিউরিটির প্রাথমিক আলোচনায় আমরা প্রথাগত Read (r), Write (w) এবং Execute (x) পারমিশন (Standard File Permissions) নিয়ে বিস্তারিত পড়াশোনা করেছি। কিন্তু এন্টারপ্রাইজ সিস্টেম অ্যাডমিনিস্ট্রেশন এবং মাল্টি-ইউজার পরিবেশের কিছু জটিল ও সংবেদনশীল কারিগরি চাহিদা মেটাতে স্ট্যান্ডার্ড পারমিশন মডেল একাকী যথেষ্ট নয়। উদাহরণস্বরূপ, একজন সাধারণ ব্যবহারকারী যখন তার নিজের অ্যাকাউন্টের পাসওয়ার্ড পরিবর্তন করতে চান, তখন ব্যাকএন্ডে তাকে এমন একটি সিস্টেম ফাইল (/etc/shadow) মডিফাই করতে হয় যা মূলত সম্পূর্ণ root প্রিভিলেজড লকড অবস্থায় থাকে।

সাধারণ ইউজারকে রুট পাসওয়ার্ড না জানিয়ে কীভাবে এই কাজগুলো নিরাপদ উপায়ে স্বয়ংক্রিয়ভাবে সম্পন্ন করা যায়? লিনাক্স কার্নেল এই ধরণের বিশেষ পরিস্থিতির জন্য স্ট্যান্ডার্ড পারমিশন লেয়ারের ঠিক উপরে আরেকটি অ্যাডভান্সড সিকিউরিটি মেকানিজম ব্যবহার করে, যাকে বলা হয় File Special Permissions। আজকের এই ডিপ-ডাইভ টিউটোরিয়ালে আমরা লিনাক্সের ৩টি স্পেশাল পারমিশন—SUID, SGID এবং Sticky Bit এর গভীর মেকানিজম, অক্টাল ভ্যালু এবং প্র্যাকটিক্যাল কনফিগারেশন নিয়ে বিস্তারিত আলোচনা করবো।

১. থ্রি স্পেশাল পারমিশন টাইপস (The Three Special Permissions)

স্ট্যান্ডার্ড পারমিশনের মতো স্পেশাল পারমিশনও মূলত ৩ ধরনের হয়ে থাকে এবং এদের প্রতিটির কাজের লজিক্যাল আর্কিটেকচার সম্পূর্ণ ভিন্ন।

ক. SUID (Set User ID)

SUID মূলত শুধুমাত্র এক্সিকিউটেবল বাইনারি ফাইলের ওপর কাজ করে। যখন কোনো ফাইলে SUID সেট করা থাকে, তখন যেকোনো সাধারণ ব্যবহারকারী ওই ফাইল বা কমান্ডটি রান করলে, ফাইলটি মেমরিতে ওই সাধারণ ইউজারের প্রিভিলেজে এক্সিকিউট না হয়ে বরং ফাইলের ওরিজিনাল Owner (Owner's Privilege)-এর ক্ষমতা নিয়ে এক্সিকিউট হয়।

  • The Classical Example: লিনাক্সের passwd কমান্ড (যা /usr/bin/passwd পাথে থাকে)। এই ফাইলটির ওনার হলো root। সাধারণ ইউজার যখন এটি রান করেন, SUID একটিভ থাকায় কমান্ডটি সাময়িকভাবে root এর ক্ষমতা লাভ করে এবং /etc/shadow ফাইলে নতুন পাসওয়ার্ড হ্যাশ রাইট করে সেশন ক্লোজ করে দেয়।
  • Visual Indicator: ফাইলের ওনার ব্লকের যেখানে x থাকার কথা, সেখানে ছোট হাতের s (যেমন- rws) দেখা যাবে।

রিয়েল-লাইফ সিনারিও (The Signatory Token Analogy)

SUID-এর কার্যপদ্ধতি সহজে বোঝার জন্য একটি প্রাতিষ্ঠানিক ডকুমেন্টের কথা চিন্তা করুন:

  • ধরি অফিসের একটি বিশেষ নোটিশে বাধ্যতামূলকভাবে কোম্পানির ম্যানেজিং ডিরেক্টরের (Root User) অফিশিয়াল গোল সিল ও দস্তখত বা সিগনেচার প্রয়োজন। আপনি একজন সাধারণ এমপ্লয়ি (Regular User) হওয়ায় আপনার নিজস্ব কোনো আইনি সিল নেই।
  • কোম্পানি একটি স্বয়ংক্রিয় স্ট্যাম্পিং মেশিন (SUID Binary) তৈরি করে ফ্রন্ট ডেস্কে রেখে দিল। আপনি ওই মেশিনের স্লটে কাগজ ঢুকিয়ে বাটন চাপলেই মেশিনটি ম্যানেজিং ডিরেক্টরের ডেডিকেটেড সিগনেচার টোকেন ব্যবহার করে কাগজে সিল মেরে দেয়। মেশিনটি রান করেছেন আপনি, কিন্তু কাগজের ওপর ক্ষমতা প্রয়োগ হয়েছে ডিরেক্টরের। এটিই হলো SUID।

খ. SGID (Set Group ID)

SGID ফাইল এবং ডিরেক্টরি—উভয়ের ওপরই অত্যন্ত চমৎকারভাবে কাজ করতে পারে:

  • ফাইলের ক্ষেত্রে: কোনো ফাইল এক্সিকিউট করার সময় এটি ফাইলের ওরিজিনাল Group Owner-এর প্রিভিলেজ নিয়ে মেমরিতে রান করে (SUID-এর মতোই, তবে এখানে ওনারের বদলে গ্রুপ প্রায়োরিটি পায়)।
  • ডিরেক্টরির ক্ষেত্রে: এটি এন্টারপ্রাইজ শেয়ারড ফোল্ডার ম্যানেজমেন্টের জন্য সবচেয়ে বেশি ব্যবহৃত হয়। কোনো ডিরেক্টরিতে SGID সেট করা থাকলে, ওই ফোল্ডারের ভেতরে যেকোনো ইউজার এসে নতুন কোনো ফাইল বা সাব-ফোল্ডার তৈরি করলে, সেই নতুন ফাইলের গ্রুপ ওনারশিপ স্বয়ংক্রিয়ভাবে ওই প্যারেন্ট ডিরেক্টরির গ্রুপ ওনারশিপ ইনহেরিট (Inherit) বা ক্লোন করে নেয়। ইউজারের নিজস্ব প্রাইমারি গ্রুপ সেখানে স্কিপ হয়ে যায়।
  • Visual Indicator: গ্রুপের পারমিশন ব্লকের যেখানে x থাকার কথা, সেখানে ছোট হাতের s (যেমন- r-s) দেখা যাবে।

গ. Sticky Bit (The Restricted Deletion Flag)

Sticky Bit মূলত শুধুমাত্র ডিরেক্টরি বা ফোল্ডারের ওপর প্রয়োগ করা হয়। এটি একটি মাল্টি-ইউজার শেয়ারড ডিরেক্টরিতে রাইট পারমিশনের বিশৃঙ্খলা বা এরর প্রতিরোধ করার জন্য একটি প্রাকৃতিক ফায়ারওয়াল।

  • কাজের মেকানিজম: একটি ফোল্ডারে যদি সবার জন্য ফুল রাইট পারমিশন (rwx) দেওয়া থাকে, তবে স্বাভাবিক নিয়মে যেকোনো ইউজার অন্য ইউজারের ফাইল ডিলিট বা রিনেম করে দিতে পারেন। কিন্তু ডিরেক্টরিতে Sticky Bit অন করে দিলে, ওই ফোল্ডারের ভেতরের কোনো ফাইল শুধুমাত্র সেই ফাইলের Original Owner অথবা সিস্টেমের Root User ছাড়া অন্য কেউ ডিলিট বা রিনেম করতে পারবেন না।
  • The Classical Example: লিনাক্সের গ্লোবাল টেম্পোরারি ডিরেক্টরি /tmp। এখানে সব ইউজার ফাইল রাইট করতে পারেন, কিন্তু কেউ কারো ফাইলে হস্তক্ষেপ করতে পারেন না।
  • Visual Indicator: আদার্স বা ওয়ার্ল্ড ব্লকের যেখানে x থাকার কথা, সেখানে একদম শেষ ক্যারেক্টারটি ছোট হাতের t (যেমন- rwt) হয়ে যাবে।

২. স্পেশাল পারমিশন গাণিতিক ক্যালকুলেশন ও ওকটেট টেবিল

স্ট্যান্ডার্ড পারমিশনের নিউমেরিক মান (Read=4, Write=2, Execute=1) হিসাব করার আগে, ৩-ডিজিটের সংখ্যার একদম শুরুতে আরেকটি ৪র্থ ডিজিট যোগ করে স্পেশাল পারমিশন গাণিতিকভাবে হিসাব করা হয়। এদের ওকটেট মান নিচে দেওয়া হলো:

Special Permission Name Numeric / Octal Value Symbolic Flag Representation Target Execution Scope
SUID 4 s (In Owner Block) Executable binary files (Changes execution context to file Owner).
SGID 2 s (In Group Block) Directories & files (Enforces group ownership inheritance).
Sticky Bit 1 t (In Others Block) Shared directories (Restricts file deletion to file Owner only).

এই মানগুলো যোগ করে আমরা একটি ৪-ডিজিটের সংখ্যা তৈরি করি। উদাহরণস্বরূপ:

  • 4755 পারমিশনের অর্থ: শুরুতে ৪ থাকায় এটি SUID একটিভ করবে, এবং বাকি অংশ ডিফল্ট rwxr-xr-x মোডে থাকবে।
  • 2770 পারমিশনের অর্থ: শুরুতে ২ থাকায় এটি ডিরেক্টরিতে SGID লক অন করবে, ওনার ও গ্রুপ ফুল এক্সেস (77) পাবে এবং আদার্স সম্পূর্ণ ব্লক (0) থাকবে।
  • 1777 পারমিশনের অর্থ: শুরুতে ১ থাকায় এটি Sticky Bit অন করবে এবং বাকি অংশ গ্লোবাল ফুল এক্সেস (rwxrwxrwx) মোডে থাকবে।

ক্রিটিক্যাল লজিক: বড় হাতের 'S' বা 'T' দেখার অর্থ কী?

অনেক সময় ls -l দেওয়ার পর পারমিশন স্ট্রিংয়ে ছোট হাতের s বা t এর পরিবর্তে বড় হাতের S বা T দেখা যায়। এটি একটি টেকনিক্যাল এরর কন্ডিশন নির্দেশ করে। এর মানে হলো, আপনি ফাইল বা ডিরেক্টরিতে স্পেশাল পারমিশন অন করেছেন ঠিকই, কিন্তু সেখানে বেসিক Execute (x) পারমিশনটি অন করতে ভুলে গেছেন। বেসিক এক্সিকিউট পারমিশন ছাড়া স্পেশাল পারমিশন মেমরিতে কাজ করতে পারে না।

৩. প্র্যাকটিক্যাল CLI অপারেশন: কনফিগারেশন এবং ভেরিফিকেশন স্টেপস

আসুন আমরা টার্মিনাল বা কমান্ড লাইনের মাধ্যমে এই স্পেশাল পারমিশনগুলো প্র্যাকটিক্যালি তৈরি, অ্যাসাইন এবং ভেরিফাই করার নিখুঁত ধাপগুলো সম্পাদন করি।

ধাপ ১: SUID পারমিশন কনফিগার করা (Numeric & Symbolic)

একটি কাস্টম বাইনারি ফাইলে SUID এনাবল করার কমান্ড:

# Numeric Method (4000 সিরিজের ওকটেট)
$ sudo chmod 4755 /usr/local/bin/custom_tool

# Symbolic Method (u+s)
$ sudo chmod u+s /usr/local/bin/custom_tool

# ভেরিফিকেশন কমান্ড
$ ls -l /usr/local/bin/custom_tool
# আউটপুট: -rwxr-xr-x -> পরিবর্তিত হয়ে হবে -> -rwsr-xr-x

ধাপ ২: এন্টারপ্রাইজ শেয়ারড ডিরেক্টরিতে SGID লক কনফিগার করা

ধরুন আপনার মিডিয়া হাউসের dev_team গ্রুপের জন্য একটি সেন্ট্রাল ফোল্ডার তৈরি করতে হবে যেখানে সবাই ফাইল শেয়ার করবেন:

# একটি নতুন শেয়ারড ডিরেক্টরি তৈরি করা
$ sudo mkdir /opt/shared_pool

# ফোল্ডারের গ্রুপ মালিকানা পরিবর্তন করা
$ sudo chown :dev_team /opt/shared_pool

# ডিরেক্টরিতে SGID লক অন করা (2000 সিরিজ বা g+s)
$ sudo chmod 2770 /opt/shared_pool

# ভেরিফিকেশন চেক
$ ls -ld /opt/shared_pool
# আউটপুট: drwxrws--- 2 root dev_team 4096 Jun 14 12:00 /opt/shared_pool

ফলাফল: এখন থেকে যেকোনো ইউজার এই ফোল্ডারের ভেতরে ফাইল বানালে তার গ্রুপ ওনারশিপ অটোমেটিকভাবে dev_team লকড হয়ে যাবে।

ধাপ ৩: ডিরেক্টরিতে Sticky Bit অন করা

একটি ফোল্ডারকে এক্সিডেন্টাল ডিলিশন থেকে রক্ষা করার জন্য স্টিকি বিট এনাবল করার মেথড:

# Symbolic Method (o+t) বা Numeric Method (1777)
$ sudo chmod +t /opt/shared_pool

# ভেরিফিকেশন চেক
$ ls -ld /opt/shared_pool
# আউটপুট: drwxrws--t 2 root dev_team 4096 Jun 14 12:00 /opt/shared_pool

৪. Troubleshooting এবং মারাত্মক নিরাপত্তা ঝুঁকি (Security Hardening)

স্পেশাল পারমিশন যেমন ইনফ্রাস্ট্রাকচারকে নমনীয়তা দেয়, তেমনি অসচেতনভাবে ব্যবহার করলে এটি হ্যাকারদের জন্য ব্যাকডোর (Backdoor) তৈরির সবচেয়ে বড় হাতিয়ারে পরিণত হতে পারে।

  • The SUID Security Threat (SUID ব্যাকডোর অ্যাটাক): সাইবার সিকিউরিটি এবং পেনিট্রেশন টেস্টিং ক্যারিয়ারে Privilege Escalation-এর জন্য হ্যাকাররা সবচেয়ে বেশি খোঁজে এই SUID ফাইল। কোনো হ্যাকার যদি সিস্টেমে ঢুকে সাময়িকভাবে সাধারণ ইউজার থেকে রুট হতে চায়, সে টেক্সট এডিটর বা বাশ শেলের বাইনারিতে SUID অন করে রেখে দিতে পারে (যেমন- chmod u+s /bin/bash)। এর ফলে পরবর্তীতে সে যেকোনো নরমাল অ্যাকাউন্ট থেকে bash -p দিলেই সরাসরি কোনো পাসওয়ার্ড ছাড়াই ওএস লেভেলের রুট সেল পেয়ে যাবে।
  • Audit and Detection (সার্ভার অডিট কমান্ড): একজন সিস্টেম অ্যাডমিনিস্ট্রেটর হিসেবে আপনার সার্ভারে অনাকাঙ্ক্ষিত কোনো SUID বা SGID ফাইল লুকিয়ে আছে কি না তা নিয়মিত নিচের find কমান্ড দিয়ে স্ক্যান করে অডিট করা উচিত: 
    # পুরো হার্ডডিস্কে স্ক্যান করে সমস্ত SUID ফাইল খুঁজে বের করা
$ sudo find / -perm /4000 -type f 2>/dev/null

# সমস্ত SGID ফাইল খুঁজে বের করার কমান্ড
$ sudo find / -perm /2000 -type f 2>/dev/null

    অ্যানালাইসিস টিপস: আউটপুটের লিস্টে যদি স্ট্যান্ডার্ড বাইনারি (যেমন passwd, sudo, ping) ছাড়া কোনো কাস্টম স্ক্রিপ্ট বা টেক্সট এডিটরের নাম দেখতে পান, তবে কালক্ষেপণ না করে সেটির পারমিশন অডিট করুন।

  • Removing Special Permissions: মেমরি বা ফাইল থেকে স্পেশাল পারমিশন সম্পূর্ণ রিমুভ বা প্রত্যাহার করার কমান্ড: 
    $ sudo chmod u-s /usr/local/bin/custom_tool
$ sudo chmod g-s /opt/shared_pool
$ sudo chmod -t /opt/shared_pool

উপসংহার (Conclusion)

Linux File Special Permissions অপারেটিং সিস্টেমের সিকিউরিটি কনটেক্সট এবং অ্যাক্সেস ম্যানেজমেন্টের এক অত্যন্ত শক্তিশালী ও জটিল কারিগরি স্তর। SUID-এর মাধ্যমে রানটাইম ওনার প্রিভিলেজ এলিভেশন, SGID-এর মাধ্যমে ডিরেক্টরি গ্রুপ ওনারশিপ ইনহেরিটেন্স এবং Sticky Bit-এর মাধ্যমে শেয়ারড ডিলিটিং প্রোটেকশন—প্রতিটি উপাদানই এন্টারপ্রাইজ ইনফ্রাস্ট্রাকচার ডিজাইনে সুনির্দিষ্ট ভূমিকা পালন করে। একজন প্রফেশনাল সিস্টেম ইঞ্জিনিয়ার হিসেবে এই অ্যাডভান্সড ফ্ল্যাগগুলোর ব্যবহার ও তাদের নিরাপত্তা ঝুঁকি (Security Implications) নিখুঁতভাবে বুঝতে পারা ক্যারিয়ারের গভীরতা বহুগুণ বাড়িয়ে দেবে। নিয়মিত সার্ভার অডিট এবং সতর্কতার সাথে ল্যাব লার্নিংই আপনাকে লিনাক্স হার্ডেনিংয়ে এক্সপার্ট করে তুলবে।
