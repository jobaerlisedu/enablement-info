Introduction, Concepts and Deep Dive into File Permissions

14 June, 2026 Linux Administration Jobaer Hossain

লিনাক্স অপারেটিং সিস্টেমের অন্যতম প্রধান শক্তি হলো এর সিকিউরিটি মডেল। এটি একটি পিওর Multi-user System হিসেবে ডিজাইন করা হয়েছে, যেখানে একই সার্ভারে বা কম্পিউটারে একসাথে শত শত ইউজার সম্পূর্ণ আইসোলেটেড অবস্থায় কাজ করতে পারেন। এই মাল্টি-ইউজার এনভায়রনমেন্টে ডাটার নিরাপত্তা এবং গোপনীয়তা বজায় রাখার মূল চাবিকাঠি হলো File Permissions। কোনো ইউজার অন্য ইউজারের ফাইল দেখতে পারবেন কি না, এডিট করতে পারবেন কি না বা কোনো প্রোগ্রাম রান করতে পারবেন কি না, তা সম্পূর্ণ এই সিস্টেমের মাধ্যমে কার্নেল লেভেলে সুনির্দিষ্টভাবে নিয়ন্ত্রিত হয়।

একজন প্রফেশনাল সিস্টেম অ্যাডমিনিস্ট্রেটর বা সাইবার সিকিউরিটি স্পেশালিস্ট হতে হলে লিনাক্স ফাইল পারমিশনের প্রতিটি খুঁটিনাটি নিখুঁতভাবে বোঝা অত্যন্ত জরুরী। পারমিশন সেটিংসের সামান্য একটি ভুল পুরো কোম্পানির সংবেদনশীল ডাটা লিক করে দিতে পারে। আজকের এই ডিপ-ডাইভ টিউটোরিয়ালে আমরা ফাইল পারমিশনের বেসিক স্ট্রাকচার, ৩টি ইউজার ক্লাস, ২টি কনফিগারেশন মেথড এবং বাস্তব জীবনের ট্রাবলশুটিং নিয়ে বিস্তারিত আলোচনা করবো।

১. থ্রি ইউজার ক্লাস (The Three User Classes)

লিনাক্সে যেকোনো ফাইল বা ডিরেক্টরি তৈরি করার সাথে সাথে ওএস তার সুরক্ষার জন্য ব্যবহারকারীদের ৩টি সুনির্দিষ্ট ক্লাসে ভাগ করে ফেলে:

  • User / Owner (u): যিনি ফাইলটি ফিজিক্যালি তৈরি করেছেন বা ফাইলের বর্তমান মালিক। ডিফল্টভাবে ফাইলের ওনারের ক্ষমতা ওই ফাইলের ওপর সবচেয়ে বেশি থাকে।
  • Group (g): লিনাক্সে একাধিক ইউজারকে একটি নির্দিষ্ট গ্রুপে (যেমন: developers বা accounting) যুক্ত করা যায়। এই ক্লাসের পারমিশনটি ওই নির্দিষ্ট গ্রুপের সমস্ত মেম্বারদের ওপর একসাথে প্রযোজ্য হয়।
  • Others / World (o): ওনার নিজে এবং গ্রুপের মেম্বাররা ছাড়া সিস্টেমের বাকি সমস্ত সাধারণ ব্যবহারকারী যারা এই ফাইলের সাথে সরাসরি যুক্ত নন, তারা এই ক্লাসের অন্তর্ভুক্ত।

রিয়েল-লাইফ সিনারিও (The Digital Office Locker Analogy)

ফাইল পারমিশনের ৩টি ক্লাস সহজে বোঝার জন্য একটি কর্পোরেট অফিসের লকার বা ড্রয়ারের কথা চিন্তা করুন:

  • User (ড্রয়ারের মালিক): আপনি নিজে ওই টেবিল এবং ড্রয়ারের মালিক। আপনার কাছে ড্রয়ার খোলার এবং ভেতরের ফাইল পরিবর্তন করার পূর্ণ স্বাধীনতা আছে।
  • Group (ডিপার্টমেন্টাল কলিগ): আপনার ফাইন্যান্স টিমের ৫ জন মেম্বার। আপনি ড্রয়ারের একটি কাস্টম চাবি তাদের দিয়ে রাখলেন যাতে তারা প্রয়োজনে ফাইন্যান্স ফাইলগুলো দেখতে পারেন, কিন্তু আইটি টিমের কেউ তা পারবে না।
  • Others (অফিসের পিয়ন বা বাইরের ভিজিটর): অফিসের বাকি সমস্ত মানুষ যারা ফাইন্যান্স টিমের অংশ নন। তারা ড্রয়ার স্পর্শও করতে পারবেন না।

২. দ্য থ্রি পারমিশন টাইপস (Read, Write, Execute)

প্রতিটি ইউজার ক্লাসের জন্য লিনাক্স ৩ ধরনের অ্যাক্সেস রাইটস বা পারমিশন টাইপ ফিক্সড করতে পারে। ফাইল এবং ডিরেক্টরির (ফোল্ডার) ক্ষেত্রে এদের কাজের মেকানিজম কিছুটা ভিন্ন হয়:

ক. Read (r)

  • ফাইলের ক্ষেত্রে: ফাইলের ভেতরের ডাটা বা কন্টেন্ট রিড বা দেখার অনুমতি দেয় (যেমন cat বা less কমান্ড দিয়ে ফাইল দেখা)।
  • ডিরেক্টরির ক্ষেত্রে: ডিরেক্টরির ভেতরে কী কী ফাইল বা সাব-ফোল্ডার জমা আছে তার লিস্ট দেখার অনুমতি দেয় (যেমন ls কমান্ড রান করা)।

খ. Write (w)

  • ফাইলের ক্ষেত্রে: ফাইলের ভেতরের টেক্সট মডিফাই, এডিট বা পরিবর্তন করার অনুমতি দেয় (যেমন vim বা nano দিয়ে এডিট করে ফাইল সেভ করা)।
  • ডিরেক্টরির ক্ষেত্রে: ওই ফোল্ডারের ভেতরে নতুন ফাইল তৈরি করা, পুরনো ফাইল ডিলিট করা বা ফাইলের নাম পরিবর্তন (Rename) করার অনুমতি দেয়।

গ. Execute (x)

  • ফাইলের ক্ষেত্রে: ফাইলটি যদি কোনো প্রোগ্রাম, বাইনারি বা Bash Script হয়ে থাকে, তবে সেটিকে মেমরিতে এক্সিকিউট বা রান করার অনুমতি দেয়।
  • ডিরেক্টরির ক্ষেত্রে: এটি অত্যন্ত গুরুত্বপূর্ণ। ডিরেক্টরির ক্ষেত্রে Execute পারমিশনের অর্থ হলো সেই ফোল্ডারের ভেতরে প্রবেশ করা বা ট্রাভার্স করার ক্ষমতা (যেমন cd কমান্ড দিয়ে ফোল্ডারে ঢোকা)। ডিরেক্টরিতে x পারমিশন না থাকলে r বা w কোনো কাজই করতে পারে না।

৩. পারমিশন স্ট্রিং এর নিখুঁত ব্যবচ্ছেদ

টার্মিনালে যখন আমরা ls -l (Long Listing) কমান্ড দিই, তখন ফাইলের নামের শুরুতে ১০টি ক্যারেক্টারের একটি স্ট্রিং দেখা যায়। আসুন এটির নিখুঁত কারিগরি পোস্টমর্টেম করি:

drwxr-xr-- 2 jobaer IT_Dept 4096 Jun 14 12:00 projects

এই ১০টি ক্যারেক্টারকে মূলত ৪টি অংশে ভাগ করে পড়তে হয়:

  1. ১ম ক্যারেক্টার (File Type Indicator): এটি ফাইলের ধরণ নির্দেশ করে। - মানে এটি একটি রেগুলার ফাইল, d মানে ডিরেক্টরি বা ফোল্ডার, এবং l মানে সিম্বলিক লিঙ্ক বা শর্টকাট।
  2. ২য় থেকে ৪র্থ ক্যარეক্টার (User Permissions): ওনারের পারমিশন ব্লক। এখানে rwx থাকার মানে ওনার রিড, রাইট এবং এক্সিকিউট করতে পারবেন।
  3. ৫ম থেকে ৭ম ক্যারেক্টার (Group Permissions): গ্রুপের পারমিশন ব্লক। এখানে r-x থাকার মানে গ্রুপের মেম্বাররা শুধু রিড এবং এক্সিকিউট করতে পারবেন, কিন্তু মডিফাই বা রাইট করতে পারবেন না (ড্যাশ - মানে পারমিশন অনুপস্থিত)।
  4. ৮ম থেকে ১০ম ক্যারেক্টার (Others Permissions): আদার্স ব্লক। এখানে r-- থাকার মানে বাইরের বাকি দুনিয়ার সবাই ফাইলটি শুধু রিড করতে পারবেন, রাইট বা এক্সিকিউট করতে পারবেন না।

৪. ফাইল পারমিশন মেজারমেন্ট ও ক্যালকুলেশন টেবিল

লিনাক্স কার্নেল ব্যাকঅ্যান্ডে পারমিশন বিটগুলোকে বাইনারি ও অক্টাল (Numeric) ভ্যালু দিয়ে প্রসেস করে। প্রতিটি পারমিশনের একটি নির্দিষ্ট গাণিতিক মান রয়েছে:

Permission Flag Binary Weight Representation Octal / Numeric Value Technical Access Capability
r (Read) 2^2 = 100 (Binary) 4 Viewing binary data or reading configuration properties.
w (Write) 2^1 = 010 (Binary) 2 Modifying structure, updating payload, or erasing blocks.
x (Execute) 2^0 = 001 (Binary) 1 Triggering process execution or crossing directory limits.
- (None) 000 (Binary) 0 No active logical access granted.

এই মানগুলো যোগ করে আমরা একটি ৩-ডিজিটের সংখ্যা বা অক্টাল পারমিশন তৈরি করি। উদাহরণস্বরূপ:

  • 755 পারমিশনের অর্থ:
    • User: $4 + 2 + 1 = 7$ (rwx)
    • Group: $4 + 0 + 1 = 5$ (r-x)
    • Others: $4 + 0 + 1 = 5$ (r-x)
  • 644 পারমিশনের অর্থ:
    • User: $4 + 2 + 0 = 6$ (rw-)
    • Group: $4 + 0 + 0 = 4$ (r--)
    • Others: $4 + 0 + 0 = 4$ (r--)

৫. প্র্যাকটিক্যাল CLI অপারেশন: chmod ও chown-এর ব্যবহার

ফাইল পারমিশন ও মালিকানা পরিবর্তন করার জন্য লিনাক্সে দুটি শক্তিশালী কমান্ড ব্যবহৃত হয়: chmod (Change Mode) এবং chown (Change Owner)। নিচে ধাপে ধাপে এদের ব্যবহার দেখানো হলো।

পদ্ধতি ১: Numeric (Absolute) Method ব্যবহার করে পারমিশন চেঞ্জ

এটি সবচেয়ে ফাস্ট এবং প্রফেশনাল মেথড। এখানে সরাসরি অক্টাল নাম্বার ব্যবহার করা হয়:

# একটি স্ক্রিপ্ট ফাইলকে ওনারের জন্য ফুল পারমিশন এবং বাকিদের জন্য রিড-অনলি করা
$ chmod 755 deployment_script.sh

# একটি অত্যন্ত গোপন ফাইলকে শুধুমাত্র ওনার ছাড়া বাকি সবার জন্য সম্পূর্ণ ব্লক (00) করা
$ chmod 600 private_key.pem

পদ্ধতি ২: Symbolic Method ব্যবহার করে পারমিশন চেঞ্জ

যদি আপনি বাকি ক্লাসের পারমিশন স্পর্শ না করে শুধুমাত্র নির্দিষ্ট কোনো ক্লাসের পারমিশন যোগ বা বিয়োগ করতে চান, তবে এটি ব্যবহার করা হয়। এর জন্য প্লাস (+), মাইনাস (-) এবং ইকুয়াল (=) অপারেটর ব্যবহার করা হয়:

# Others ক্লাসের জন্য এক্সিকিউট পারমিশন যোগ করা
$ chmod o+x script.sh

# Group ক্লাসের জন্য রাইট পারমিশন কেড়ে নেওয়া বা বিয়োগ করা
$ chmod g-w documentation.docx

# ওনারের জন্য পারমিশন একদম ফিক্সড (rw-) করে দেওয়া
$ chmod u=rw data.txt

পদ্ধতি ৩: ফাইলের ওনার এবং গ্রুপ মালিকানা পরিবর্তন (chown)

কোনো ফাইলের মালিকানা পরিবর্তন করতে chown কমান্ড রুট বা sudo প্রিভিলেজ নিয়ে রান করতে হয়:

# ফাইলের ওনার পরিবর্তন করে ইউজার 'jobaer' কে দেওয়া
$ sudo chown jobaer report.pdf

# একই সাথে ওনার 'jobaer' এবং গ্রুপ 'IT_Dept' সেট করার স্ট্যান্ডার্ড মেথড
$ sudo chown jobaer:IT_Dept web_configs.conf

# একটি ডিরেক্টরি এবং তার ভেতরের সমস্ত ফাইলের মালিকানা রিকার্সিভলি (-R) চে่ง করা
$ sudo chown -R jobaer:developers /var/www/html/app

৬. Troubleshooting ও সাধারণ পারমিশন চ্যালেঞ্জসমূহ

  • The "Permission Denied" Error on Scripts: আপনি একটি বাশ স্ক্রিপ্ট তৈরি করলেন এবং রুট ইউজার হিসেবে থাকার পরও ./backup.sh দিতেই ওএস এরর দিল "Permission Denied"।
    কারণ: লিনাক্সে কোনো নতুন ফাইল তৈরি হলে ডিফল্টভাবে সিকিউরিটির জন্য তার Execute (x) পারমিশন বন্ধ থাকে। রুট ইউজারও এক্সিকিউট ফ্ল্যাগ ছাড়া স্ক্রিপ্ট সরাসরি রান করতে পারেন না।
    সমাধান: ফাইলটিতে রান করার অনুমতি যোগ করুন: 
    $ chmod +x backup.sh
  • Web Server 403 Forbidden Error: Nginx বা Apache ওয়েব সার্ভার ডিপ্লয় করার পর ব্রাউজারে সাইট লোড করলে "403 Forbidden" এরর আসে।
    কারণ: ওয়েব সার্ভারের নিজস্ব ডেমো ইউজার অ্যাকাউন্টটি (যেমন www-data বা nginx) আপনার ডিরেক্টরি ট্রির ভেতরে ঢোকার বা রিড করার অনুমতি পাচ্ছে না। ফোল্ডারে x পারমিশন মিসিং থাকলে এটি হয়।
    সমাধান: নিশ্চিত করুন যে আপনার ওয়েব ডিরেক্টরির পারমিশন 755 এবং ফাইলের পারমিশন 644 আছে এবং তাদের প্যারেন্ট ফোল্ডারগুলোতেও অ্যাক্সেস গ্র্যান্টেড আছে: 
    $ sudo find /var/www/html/ -type d -exec chmod 755 {} \;
$ sudo find /var/www/html/ -type f -exec chmod 644 {} \;
  • The Dangerous `chmod 777` Pitfall: অনেক সময় জুনিয়র এডমিনরা কোনো পারমিশন এরর পেলেই ট্রাবলশুটিং এড়াতে শর্টকাট হিসেবে chmod 777 filename বা chmod -R 777 /var/www মেরে দেন। এটি একটি মারাত্মক Security Vulnerability। 777 এর অর্থ হলো নেটওয়ার্কের যেকোনো সাধারণ ইউজার বা হ্যাকার আপনার ফাইলে যা ইচ্ছা তাই লিখতে পারবে এবং ম্যালিসিয়াস কোড ইনজেক্ট করে পুরো সার্ভার হ্যাক করে নিতে পারবে। প্রোডাকশন সার্ভারে 777 ব্যবহার সম্পূর্ণ নিষিদ্ধ।

উপসংহার (Conclusion)

Linux File Permissions অপারেটিং সিস্টেমের নিরাপত্তার প্রথম এবং সবচেয়ে শক্তিশালী লজিক্যাল ডিফেন্স লাইন। ৩টি ইউজার ক্লাসের (User, Group, Others) এবং ৩টি অ্যাক্সেস টাইপের (Read, Write, Execute) এই নিখুঁত গাণিতিক বিন্যাসই লিনাক্সকে দেয় চরম স্থায়িত্ব ও করাপশন-মুক্ত কর্মপরিবেশ। একজন দক্ষ সিস্টেম অ্যাডমিনিস্ট্রেটর হিসেবে chmod-এর অক্টাল ভ্যালু এবং chown-এর রিকার্সিভ অপারেশনগুলোর ওপর নিখুঁত নিয়ন্ত্রণ থাকা ক্যারিয়ারের জন্য অত্যন্ত ফলপ্রসূ। এটি আপনার ডেভঅপ্স আর্কিটেকচার এবং সার্ভার হার্ডেনিংকে রাখবে সম্পূর্ণ নিরাপদ ও এরর-মুক্ত। নিয়মিত ল্যাব প্র্যাকটিস এবং প্যাকেট/ফাইল অডিট করার অভ্যাসই আপনাকে এই কারিগরিতে মাস্টার করে তুলবে।
