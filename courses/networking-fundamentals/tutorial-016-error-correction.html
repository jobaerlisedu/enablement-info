Error Correction in Computer Networks: Detailed Guide

15 May, 2026 Network Engineering Jobaer Hossain

পূর্ববর্তী আর্টিকেলে আমরা জেনেছি কীভাবে Error Detection এর মাধ্যমে রিসিভার বুঝতে পারে যে ডাটা ট্রান্সমিশনের সময় কোনো বিট (Bit) পরিবর্তিত বা নষ্ট (Corrupted) হয়েছে কি না। কিন্তু শুধু ভুল ধরলেই তো হবে না, সেই ভুলটি তো ঠিকও করতে হবে! ডাটার ভুলটি খুঁজে বের করে সেটিকে আবার আগের সঠিক অবস্থায় ফিরিয়ে আনার এই জাদুকরী প্রক্রিয়াটিকেই বলা হয় Error Correction।

আজকের এই ডিপ-ডাইভ টিউটোরিয়ালে আমরা শিখবো Error Correction কী, এটি কীভাবে কাজ করে, ARQ এবং Forward Error Correction (FEC) এর পার্থক্য এবং নেটওয়ার্ক ট্রাবলশুটিংয়ে এর প্রভাব।

Error Correction কী? (What is Error Correction?)

Error Correction হলো এমন একটি মেকানিজম, যার মাধ্যমে রিসিভার শুধু ডাটার ভুল ধরতেই পারে না, বরং ঠিক কোন জায়গায় ভুল হয়েছে তা শনাক্ত করে সেটিকে নিজে নিজে অথবা সেন্ডারের সাহায্যে ঠিক করে নিতে পারে। এটি মূলত OSI Model-এর Data Link Layer (Layer 2) এবং Transport Layer (Layer 4) এ কাজ করে।

রিয়েল-লাইফ উদাহরণ: আপনি ফোনে আপনার বন্ধুর সাথে কথা বলছেন। লাইনে প্রচুর নয়েজ (Noise) থাকার কারণে আপনি তার একটি কথা ঠিকমতো শুনতে পেলেন না। এখন আপনি দুটি কাজ করতে পারেন: ১. বন্ধুকে বলতে পারেন কথাটি আবার রিপিট করতে (একে বলে Retransmission), অথবা ২. কথার প্রসঙ্গের ওপর ভিত্তি করে আপনি নিজেই গেস (Guess) করে নিতে পারেন সে কী বলেছে (একে বলে Forward Error Correction)।

১. Error Correction এর প্রধান দুটি পদ্ধতি

নেটওয়ার্কিংয়ে ডাটা কারেকশন করার জন্য মূলত দুটি অ্যাপ্রোচ বা পদ্ধতি ব্যবহার করা হয়:

A. Backward Error Correction (Retransmission বা ARQ)

আমাদের সাধারণ ইন্টারনেট এবং লোকাল এরিয়া নেটওয়ার্কে (LAN) এটি সবচেয়ে বেশি ব্যবহৃত হয়। এই পদ্ধতিতে রিসিভার যদি দেখে ডাটাতে কোনো Error আছে, তবে সে ওই নষ্ট ডাটাটি (Frame) ড্রপ করে দেয় এবং সেন্ডারকে বলে ডাটাটি আবার পাঠাতে। এই প্রক্রিয়াটিকে ARQ (Automatic Repeat Request) বলা হয়।

ARQ মূলত তিনটি মেকানিজমে কাজ করে:

  • Stop-and-Wait ARQ: সেন্ডার একটি ডাটা ফ্রেম পাঠিয়ে অপেক্ষা করে। রিসিভার সেটি সফলভাবে পেলে একটি Acknowledgment (ACK) পাঠায়। ACK পাওয়ার পরই সেন্ডার পরবর্তী ফ্রেম পাঠায়। এটি অত্যন্ত ধীরগতির।
  • Go-Back-N ARQ: সেন্ডার একসাথে অনেকগুলো ফ্রেম পাঠাতে থাকে (যাকে Sliding Window বলে)। যদি মাঝখানের কোনো একটি ফ্রেম নষ্ট হয়, তবে রিসিভার সেই ফ্রেমের পর থেকে আর কোনো ফ্রেম গ্রহণ করে না। সেন্ডারকে তখন ওই নষ্ট ফ্রেম থেকে শুরু করে পরবর্তী সব ফ্রেম আবার পাঠাতে হয়।
  • Selective Repeat ARQ: এটি সবচেয়ে স্মার্ট। একসাথে অনেকগুলো ফ্রেম পাঠানো হয়, কিন্তু যদি মাঝখানের কোনো একটি ফ্রেম নষ্ট হয়, তবে সেন্ডার শুধুমাত্র ওই নির্দিষ্ট নষ্ট ফ্রেমটিই আবার পাঠায়। বাকিগুলো রিসিভার বাফার (Buffer) করে রাখে। TCP (Transmission Control Protocol) এই ধরনের মেকানিজম ব্যবহার করে।

B. Forward Error Correction (FEC)

এই পদ্ধতিতে রিসিভার সেন্ডারকে ডাটা পুনরায় পাঠাতে বলে না (No Retransmission)। সেন্ডার মূল ডাটার সাথে কিছু অতিরিক্ত গাণিতিক বিট (Redundant Bits) এমনভাবে সাজিয়ে পাঠায়, যাতে রিসিভার ডাটা পাওয়ার পর নিজে নিজেই হিসাব করে বের করতে পারে ঠিক কোন বিটে ভুল হয়েছে এবং সেটিকে 0 থেকে 1 বা 1 থেকে 0 তে বদলে ঠিক করে ফেলে।

  • কোথায় ব্যবহার করা হয়? যেখানে ডাটা আবার পাঠানোর সুযোগ নেই বা আবার পাঠালে অনেক সময় লাগবে। যেমন: Satellite Communication, স্পেস এক্সপ্লোরেশন, লাইভ ভিডিও স্ট্রিমিং এবং অত্যন্ত নয়েজি ওয়্যারলেস নেটওয়ার্কে।

২. Hamming Code (ফরোয়ার্ড এরর কারেকশনের জাদুকরী পদ্ধতি)

Forward Error Correction-এর সবচেয়ে জনপ্রিয় পদ্ধতি হলো Hamming Code। এটি রিচার্ড হ্যামিং ১৯৫০ সালে আবিষ্কার করেন। এটি মূলত Single-bit Error (একটি বিটের ভুল) খুঁজে বের করতে এবং ঠিক করতে পারে।

  • কীভাবে কাজ করে? সেন্ডার মূল ডাটা বিটের (Data Bits) নির্দিষ্ট কিছু পজিশনে (যেমন ১, ২, ৪, ৮ নম্বর পজিশনে) প্যারিটি বিট (Parity Bits) যুক্ত করে।
  • রিসিভার ডাটা পাওয়ার পর প্রতিটি প্যারিটি বিটের হিসাব আবার মেলায়। যদি কোনো হিসাবে গড়মিল পাওয়া যায়, তবে রিসিভার ওই প্যারিটি বিটগুলোর পজিশন নম্বর যোগ করে।
  • যোগফল যা হয়, ঠিক সেই নম্বর বিটটিতেই Error হয়েছে বলে ধরে নেওয়া হয়! রিসিভার তখন ওই বিটটিকে উল্টে দেয় (0 থাকলে 1, 1 থাকলে 0 করে দেয়)। ব্যাস, ডাটা কারেক্ট হয়ে গেল!

৩. ARQ বনাম FEC (কোনটি কোথায় ভালো?)

বৈশিষ্ট্য (Feature) ARQ (Retransmission) FEC (Forward Error Correction)
কাজের ধরন ভুল পেলে ডাটা আবার পাঠাতে বলে। ভুল পেলে নিজেই গাণিতিক হিসাবে ঠিক করে নেয়।
ব্যান্ডউইথ ব্যবহার নরমাল অবস্থায় কম, কিন্তু ভুল হলে বারবার পাঠাতে গিয়ে Bandwidth নষ্ট হয়। প্রথম থেকেই মূল ডাটার সাথে অনেক এক্সট্রা বিট পাঠাতে হয়, তাই Bandwidth বেশি লাগে।
বিলম্ব (Latency/Delay) ডাটা আবার পাঠাতে হয় বলে Delay অনেক বেশি হয়। Delay একেবারেই কম, রিয়েল-টাইম ডাটা প্রসেস হয়।
ব্যবহার ক্ষেত্র (Use Case) Ethernet LAN, Wi-Fi, File Download, TCP connections. Satellite links, Deep Space Comm, Live Video/Audio Broadcast.

৪. Cisco CLI: Network Retransmission ও Error ট্রাবলশুটিং

বাস্তব জীবনে নেটওয়ার্ক ইঞ্জিনিয়ার হিসেবে আপনাকে দেখতে হবে নেটওয়ার্কে অনেক বেশি Error বা Retransmission হচ্ছে কি না। কারণ অতিরিক্ত Retransmission নেটওয়ার্কের স্পিড স্লো করে দেয়।

ইন্টারফেস এরর চেক করা:

Data Link Layer-এ ফ্রেম ড্রপ বা এরর হচ্ছে কি না তা দেখতে:

Switch# show interfaces gigabitEthernet 0/1

আউটপুট এনালাইসিস: এখানে input errors, CRC, এবং frame এররগুলো খেয়াল করুন। যদি এগুলো বাড়তে থাকে, তবে বুঝতে হবে ফিজিক্যাল ক্যাবল বা ডুপ্লেক্স মোডে সমস্যা আছে। সুইচ এই নষ্ট ফ্রেমগুলোকে ড্রপ করছে এবং Transport Layer (TCP) বাধ্য হচ্ছে ডাটাগুলোকে বারবার Retransmit করতে।

TCP Retransmission কনসেপ্ট:

সুইচের পোর্টে কোনো কনফিগারেশন এরর বা যানজট (Congestion) থাকলে ডাটা প্যাকেট ড্রপ হয়। তখন ইউজারের পিসির TCP প্রোটোকল Selective Repeat ARQ ব্যবহার করে ওই ড্রপ হওয়া প্যাকেটটি আবার পাঠায়। এটি চেক করার জন্য সাধারণত Wireshark এর মতো প্যাকেট স্নিফিং টুল ব্যবহার করতে হয়। Wireshark-এ tcp.analysis.retransmission ফিল্টার দিয়ে খুব সহজেই দেখা যায় নেটওয়ার্কে কী পরিমাণ প্যাকেট পুনরায় পাঠানো হচ্ছে।

উপসংহার (Conclusion)

Error Correction প্রযুক্তি না থাকলে আমাদের প্রতিদিনের ইন্টারনেট ব্রাউজিং, ফাইল ডাউনলোড বা ভয়েস কলিং এক বিরাট বিশৃঙ্খলায় পরিণত হতো। লোকাল নেটওয়ার্ক এবং ইন্টারনেটে আমরা ARQ (Retransmission) এর ওপর নির্ভর করি, কারণ ফাইবার বা কপার ক্যাবলে Error হওয়ার হার অনেক কম। অন্যদিকে স্যাটেলাইট বা মহাশূন্যের মতো দুর্গম এবং নয়েজি মিডিয়ায় Forward Error Correction (FEC) বা Hamming Code এর কোনো বিকল্প নেই। এই দুটি মেকানিজম কীভাবে কাজ করে তা পরিষ্কারভাবে বুঝলে একজন নেটওয়ার্ক ইঞ্জিনিয়ার নেটওয়ার্কের লেটেন্সি (Latency) এবং পারফরম্যান্স ইস্যুগুলো খুব সহজেই সমাধান করতে পারবেন।
